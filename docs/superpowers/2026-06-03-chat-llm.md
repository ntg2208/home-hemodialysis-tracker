# Chat + LLM — On-Device Gemini with Knowledge Base

#type/build #status/planned #effort/weekend #domain/llm

Combined spec + implementation plan. Phase B of the [[Home HD Knowledge Base and Tracking System]] roadmap: wire the chat assistant to Gemini (AI Studio), backed by a user-editable knowledge base that feeds the system prompt.

## Architecture

```
Flutter (on-device)
  ├─ ChatContextBuilder  ← reads from local caches, builds system prompt
  ├─ GeminiClient        ← HTTP POST to generativeai.googleapis.com, SSE stream
  ├─ ChatController      ← manages modes (active/history/viewing), delegates to responder
  └─ KbStore             ← Firestore kb_entries, read/write from Flutter

No Cloud Run involvement. /api/chat stays as-is (placeholder).
```

**Provider:** Gemini 2.5 Flash via AI Studio (`generativelanguage.googleapis.com`). Free tier covers single-user chat.

**API key:** Stored in `flutter_secure_storage` (`ai_studio_key`). Entered once in Settings. Never sent to Cloud Run.

**Streaming — enabled via official package:** Use the `google_generative_ai` Dart package (Google's official SDK). It handles SSE parsing internally and exposes `generateContentStream()` as a `Stream<GenerateContentResponse>` — works on both web and mobile. This requires changing `ChatResponder` from `Future<String>` to `Stream<String>`, and the controller from `await reply()` to `listen()` appending chunks. The `_ThinkingDots` are replaced by live text as it arrives.

**Swap-ready:** `ChatResponder` interface already exists. `GeminiChatResponder` implements it. Swap to Claude or on-device model later by swapping the provider override.

## Knowledge Base Screen

Replaces the current `/kb` placeholder (`PlaceholderScreen`).

### Data Model

Firestore collection `kb_entries/{entry_id}`:

| Field | Type | Notes |
|---|---|---|
| `id` | string | auto-generated |
| `title` | string | e.g. "Dry Weight", "Vascular Access" |
| `content` | string | free text, markdown |
| `source` | `'user'` \| `'ai-proposed'` | tracks who created it |
| `created_at` | timestamp | |
| `updated_at` | timestamp | |

Firestore rules: same UID lock as `treatment_*` collections. Direct client access (no Cloud Run proxy).

### Screen Layout

- AppBar: "Knowledge Base"
- Body: scrollable list of entry cards
  - Each card: title (bold) + content preview (2 lines, muted)
  - Tap to edit (inline or bottom sheet)
  - Swipe to delete (with confirmation)
- FAB: "+" to add new entry
- Pull-to-refresh from Firestore

### LLM-Proposed Updates

When the assistant suggests a KB change, it includes a structured flag in the response:

```json
{
  "text": "Got it. I can update your KB:\nDry Weight: 59 kg → 58.5 kg",
  "kb_update": {
    "title": "Dry Weight",
    "content": "58.5 kg — target post-dialysis weight",
    "previous_content": "59 kg — target post-dialysis weight"
  }
}
```

Flutter renders a tappable "Update KB" chip below the assistant bubble. Tapping:
1. Opens confirmation: "Update 'Dry Weight' from 59 → 58.5?"
2. On confirm: writes to Firestore `kb_entries`
3. On cancel: dismissed, no write

The system prompt instructs the model to include `kb_update` when the user shares new information worth remembering. The model never writes directly — user always confirms.

### Tagging

KB entries use flat tags from the vault schema: `#type/knowledge-base #status/active #domain/health`.

## System Prompt Assembly

Happens in `ChatContextBuilder` — a Dart class that takes injected data (KB entries, last session, blood test rows, fitness summary, inventory stock) and returns a formatted string synchronously. Data is fetched by the caller before constructing the builder, so the builder itself has no async I/O and is straightforward to unit test with fixture data.

### Template

```
You are a personal health assistant for a home hemodialysis patient.

--- PATIENT KNOWLEDGE (user-curated) ---
{KB entries, one per line: "Title: content"}

--- CURRENT STATE (auto-assembled) ---
Last dialysis: {date}, pre {bp}, post {bp}, UF {L}, duration {min}
BP trend (7 days): pre-sys avg {n} ({change} from prior week)
Latest bloods ({date}): {selected markers with values + units}
Resting HR (7-day avg): {n} bpm. HRV (deep sleep RMSSD): {n}ms (personal 30-day baseline: {n}ms).
Sleep (7-day avg): {hours}h {min}m. Deep: {min}m, REM: {min}m.
Inventory: {critical items with stock levels}. Next delivery: {date}.

--- INSTRUCTIONS ---
- Answer concisely. Use markdown for tables and lists.
- When the user tells you something worth remembering (a new dry weight, a medication change, a symptom note), include a kb_update in your response.
- For blood test values, include reference ranges when you know them.
- If asked about something not in the current state, say so — don't guess.
- Do not give medical advice. Summarise trends and flag patterns, but always defer to the clinical team.
```

### Data Sources

| Data | Source | Method |
|---|---|---|
| KB entries | Firestore `kb_entries` | `KbStore.getAll()` |
| Last session + BP trend | Firestore `treatment_sessions` + `treatment_readings` | `TreatmentRepo` (already in Flutter) |
| Latest bloods | `BtStore` cache (local) | `BtStore.read()` |
| Resting HR, HRV, sleep | `CacheStore` (fitness summary) | `CacheStore.read('fitness_summary')` |
| Inventory | Firestore `inventory_events` | Existing inventory providers |

### Token Budget

Aim for ~800–1200 tokens for the system prompt. If KB grows large, include all entries but truncate content to ~100 chars each. The live data summary is fixed size.

|---|---|---|

| ON | set | ✓ AI assistant ready |

### Behaviour Rules

- **Toggle OFF → ON**: if key exists in storage, immediately ready. If not, show "key required" state.
- **Toggle ON → OFF**: key stays in storage. Chat shows "disabled" message.
- **Clear key**: confirmation dialog → remove from storage. Toggle stays ON, falls to "key required" state.
- **Key paste**: validate looks like an AI Studio key (starts with `AIza`), auto-fill.
- **Key stored as**: `flutter_secure_storage` key `ai_studio_key`. Toggle stored as `ai_enabled` (`'true'` / `'false'`).

## Gemini Client (`gemini_client.dart`)

### Package

```yaml
# flutter/pubspec.yaml
dependencies:
  google_generative_ai: ^0.4.0   # Google's official Dart SDK
```

No raw HTTP / dio involved. The package handles auth, request serialisation, SSE parsing, and web/mobile differences internally.

### Usage

```dart
final model = GenerativeModel(
  model: 'gemini-2.5-flash',
  apiKey: aiStudioKey,
  systemInstruction: Content.system(systemPrompt),
  generationConfig: GenerationConfig(temperature: 0.4, maxOutputTokens: 1024),
);

// Convert history to SDK format
final history = previousMessages
    .map((m) => Content(m.role == ChatRole.user ? 'user' : 'model',
        [TextPart(m.text)]))
    .toList();

final chat = model.startChat(history: history);

// Streaming — works on web + mobile
final stream = chat.sendMessageStream(Content.text(userMessage));
await for (final chunk in stream) {
  final text = chunk.text;
  if (text != null) yield text; // ChatResponder returns Stream<String>
}
```

### Interface Change

`ChatResponder` changes from `Future<String>` to `Stream<String>`:

```dart
abstract class ChatResponder {
  Stream<String> reply(String prompt, List<ChatMessage> history);
}
```

`ChatController.send()` listens to the stream and appends each chunk to the current assistant bubble in state, replacing `_ThinkingDots` on first chunk.

### Response Parsing

`kb_update` is parsed from the final accumulated text after the stream closes. System prompt instructs the model to end with `<!--KB_UPDATE {"title":"...","content":"..."}-->` if proposing a KB change. Parsed with a trailing regex. If absent, no KB update.

### Error Handling

| Error | User sees |
|---|---|
| Network error (DioException) | "Could not reach AI service. Check your connection." + Retry button |
| 401/403 (invalid key) | "API key rejected. Check your key in Settings." + link to Settings |
| 429 (rate limit) | "Too many requests. Try again in a moment." |
| 5xx (Gemini error) | "AI service error. Try again." + Retry button |
| SSE stream interrupted | Show partial response + "Response interrupted." |

## Firestore Schema — Chat History

Collection `chat_conversations/{conversation_id}`:

| Field | Type | Notes |
|---|---|---|
| `id` | string | auto-generated (uuid) |
| `title` | string | first user message, truncated to 60 chars |
| `created_at` | timestamp | |
| `updated_at` | timestamp | |
| `messages` | array of `{role, text, timestamp}` | `role`: `'user'` \| `'assistant'` |

Firestore rules: `chat_conversations` collection locked to `uid == 'homehd-treatment'` (same as `treatment_*`).

**Firebase auth precondition:** `TreatmentAuth.ensure()` is currently only called from the Treatment route's `initState`. KB and Chat both need an authenticated Firestore session. Fix: call `TreatmentAuth.ensure()` in `KbStore` and `ChatController` before any Firestore read/write, the same way `TreatmentFlow` does — the fast path (`authStateReady()` + fresh token) returns in <100ms on subsequent calls so there's no visible delay.

### Auto-Save

- **On each assistant reply**: conversation written to Firestore (upsert). Title set from first user message if not yet set.
- **On close**: save. Empty conversations (no messages) not persisted.
- **On "New chat"**: save current conversation, then fresh state.
- **History ordering**: by `updated_at` descending. Limit to last 50 conversations.

## Out of Scope

- **Multi-turn tool use** — the LLM cannot query data itself. Context is pre-assembled.
- **On-device model (Phase C)** — still using cloud API. Swap when Flutter/MediaPipe integration matures.
- **Multi-device KB sync conflicts** — single user, last-write-wins is fine.
- **Conversation search** — history list only, no full-text search.
- **Chat in Treatment screens** — chat remains the FAB bottom sheet, not embedded in session flow.
- **Scheduled summaries** — no push notifications or cron-triggered reports.
