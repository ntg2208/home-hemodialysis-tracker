How should the command layer be wired?

Three ways to connect Gemini's tool calls to Flutter app state

Data flow
💬 Chat sends prompt to Gemini
↓
Gemini calls a tool: navigate_to, set_filter, prefill_form…
↓
CommandBus (Riverpod stream) receives AppCommand
↓ ↓ ↓
Router
BloodTests filter
Treatment form

A — Command Bus (recommended)

Gemini tool use → typed AppCommand sealed class → Riverpod stream bus → each screen reacts independently. Chat and UI are fully decoupled. Clean, testable, easy to extend.

Pros

  • Screens don't know about chat
  • Add new commands without touching ChatController
  • Testable in isolation

Cons

  • Slightly more boilerplate upfront
Data flow
💬 Chat sends prompt to Gemini
↓
Gemini tool call returned
↓
ChatController calls router, providers directly
Router.go()
ref.read(btProvider)
ref.read(treatmentProvider)

B — Direct state calls

ChatController directly calls GoRouter and feature Riverpod providers. Simpler to get started, less code initially — but ChatController ends up knowing about every feature in the app.

Pros

  • Less boilerplate, faster to build

Cons

  • ChatController coupled to all features
  • Hard to test
  • Gets messy as commands grow
Data flow
💬 Chat — Gemini returns JSON in response text
↓
ChatController regex-parses JSON action from text
↓
Router.go('/blood-tests?marker=Hb&months=3')
↓
Screens read query params on build

C — JSON in response + deep links

Gemini embeds JSON action tags in its response text (like the existing KB update pattern). ChatController parses them and fires deep link navigation with query params. No tool use, no formal type system.

Pros

  • Reuses the existing KB-update pattern
  • No Gemini tool use setup

Cons

  • Fragile (string parsing)
  • Complex state (form prefill) doesn't map to URL params cleanly
  • Gemini may not reliably emit the JSON format
